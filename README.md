OSS trend dashboard hosted on GitHub Pages.
Data is generated by GitHub Actions and published as JSON snapshots.

## 1. What this project does

- Tracks repository activity changes for 7D, 14D, 30D windows
- Builds shared-contributor network edges between repositories
- Serves interactive chart/table/network UI from published snapshot files

## 2. Current architecture

## 4. Data outputs

- `meta.json`: build metadata
- `trend_7d/14d/30d.json`: ranked trend rows by window
- `network_7d/14d/30d.json`: network nodes/edges by window
- `top_repos.json`: top rows snapshot for quick usage

## 5. Metric definitions

- `Activity Œî` = current-window events - previous-window events
- `Contributor Œî` = current-window unique contributors - previous-window unique contributors
- `Trend Score` = `Activity Œî + 2 * Contributor Œî`
- `shared_contributor_count` = count of contributors active in both repositories for the selected window

## 6. Local usage

```bash
cp .env.example .env
source .env

make build-data
make run-site
```

Local URL: `http://127.0.0.1:8080`

## 7. GitHub Actions data refresh

Workflow: `.github/workflows/pages.yml`

Triggers:

- Daily schedule (UTC)
- Manual run (`workflow_dispatch`)
- Push to `main` for docs/script/workflow changes

Each run:

1. Calls GitHub API
2. Rebuilds `docs/data/*.json`
3. Deploys `docs/` to GitHub Pages

## 8. Configuration

Set via `.env` (local) or workflow env:

- `GITHUB_TOKEN` (recommended)
- `MAX_REPOS`
- `TREND_TOP_N`
- `NETWORK_MAX_EDGES`
- `MIN_SHARED_COUNT`
- `REQUEST_SLEEP_MS`
- `HTTP_TIMEOUT_SECONDS`
- `EVENT_MAX_PAGES`

## 9. Known limitations

- Network quality depends on GitHub API rate limits and available event history
- Without token, API limits are low and some repos can be skipped
- Snapshots are batch-based, not real-time
